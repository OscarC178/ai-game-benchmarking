The Opus Paradox: Why the Best Planner Produces the Worst Results

🔴 The Headline

Opus-planned games score 4.90/10 — dead last. The control group with zero planning scores 6.59. That's a +1.69 advantage for doing nothing.

📊 The Evidence: Opus vs No Planner, by Game

GameOpus PlanNo PlanΔ

🔑 Root Cause: Over-Specification

Opus produces the second-longest specs at 503 lines/game on average. But it's not just length — it's precision.

Opus specs contain exact pixel coordinates, specific hex colors, precise physics constants, and detailed state machine definitions. A 605-line Donkey Kong spec with exact girder coordinates like (40, 620) → (632, 592) leaves no room for interpretation.

Compare Gemini Pro's 93-line Donkey Kong spec: high-level description of slopes, behavior rules, rough sizes. Gemini Pro's builds score 4.0 vs Opus's 2.4 on this game.

🧩 Why Precision Hurts

When a spec says "girder from (56, 172) to (512, 186)", a builder has to implement exact slope collision detection. If it gets the math even slightly wrong — wrong interpolation, off-by-one pixel, gravity fighting the slope — the entire game breaks. The barrel physics cascade into unplayability.

When a spec says "girders slope slightly, barrels roll downhill" — the builder uses its own reasonable approach. It might use simpler collision, approximate physics, or fall back to well-tested patterns. The result is less faithful but more likely to work.

🎯 The Weak Builder Amplifier

Opus specs hurt weak builders much more than strong ones:

Builder TierWith Opus PlanNo PlanΔ
Strong (Opus, Sonnet, GPT-5.4)5.876.67+0.80
Weak (Haiku, GPT-5.4 Mini, Flash)3.376.47+3.10

Weak builders go from near-passable (6.47) to catastrophic (3.37) when given Opus specs. The detailed requirements overwhelm their context window and implementation capacity. Without specs, they build simple-but-working games. With Opus specs, they attempt complex implementations and fail.

💡 The Takeaway

Planning quality ≠ build quality. Opus writes the most thorough, detailed, architecturally sound specs — which is exactly why they produce worse games. The specs are optimized for completeness, not for the builder's ability to execute.

The best planning approach would be adaptive: detailed specs for strong builders, minimal specs for weak ones. Or, as the control group suggests: just let the builder figure it out.

This mirrors a real-world pattern: the best technical architects often write specs that only other senior engineers can implement. Junior developers do better with less prescription and more freedom.

The GLM-5 Exception

📝 GLM-5: Longest Specs, Not Worst Results

GLM-5 writes the longest specs at 615 lines/game — longer than Opus — but scores 5.59 vs Opus's 4.90. Why?

GLM-5 specs tend toward verbose description rather than exact specification. More words doesn't mean more constraint. Opus specs are dense with coordinates and constants; GLM-5 specs are fluffy with context and rationale. Builders can filter the fluff but struggle with wrong coordinates.

What goes wrong when games fail?

Reading the Bug Types

  • FRAME_FROZEN: Canvas pixels identical between screenshots — game may be stuck on title, not responding to input, or animation stopped
  • CANVAS_BLANK: All sampled pixels are black — game may not be rendering to canvas
  • ALL_FRAMES_IDENTICAL: Every screenshot looks the same — game is definitively frozen
  • PLAYTEST_TIMEOUT: Build hung during testing (Playwright timeout)

Note: 1-2 FRAME_FROZEN hits can be normal (title screens, ready screens). The scoring accounts for this.
