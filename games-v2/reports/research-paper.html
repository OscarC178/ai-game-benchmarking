Building Games with AI

What 580 Builds Taught Us About Planning, Models, and the Limits of Specification

Benchmark Rounds

Abstract. We ran two large-scale benchmarks to understand how well AI models can plan and build complete, playable games from scratch. Round 1 tested whether different builder models perform better when given the same plan — 182 builds across 9 models. Round 2 tested whether planning quality actually matters — 400 builds across 5 different plan authors and 8 builder models. The results surprised us: the model that plans best produces the worst builds, static code review dramatically overestimates playability, and the cheapest model offers the best return on investment.

"The best planner produces the worst results. Opus-planned games score 4.90 — dead last. The control group with zero planning scores 6.59."

— The Opus Paradox, Round 2

Part 1 — The Experiment: How to Benchmark AI Game Builders

1.1 The Task

We gave AI models one job: build a complete, playable vintage arcade game as a single self-contained HTML5 Canvas file. No external dependencies, no CDN scripts, no frameworks — just one .html file that opens in a browser and runs.

We chose vintage arcade games because they're well-specified, historically validated, and have clear win/lose conditions. Every game is a black-and-white decision — either the game works or it doesn't.

1.3 The Tooling

Built on OpenClaw — an agent framework that runs AI models as autonomous coding agents with isolated sessions per task. Each build ran in its own session to prevent cross-contamination. Builds were verified by automated browser playtesting with Playwright — the agent literally played every game and measured whether mechanics worked.

Round 1 Methodology (182 builds)

Two spec authors — Opus 4.6 (verbose, pixel-precise) and GPT-5.4 (concise, prose) — each filled in the same 12-section template. This was later identified as a critical flaw.

Three-judge static QA panel — GPT-5.4, Opus, and Gemini Flash each reviewed code and screenshots and assigned scores across 5 dimensions: Functionality, Keyboard Controls, Visual Fidelity, Playability, Error-Free Execution.

8 runs, 9 models, 10 games.

Round 2 Methodology (400 builds)

No template. Each planner received only this brief:

You are writing a game specification for a skilled developer who will implement {game_name} as a single self-contained HTML5 Canvas file with no external dependencies. The developer is skilled but has never played this game. Write them a spec that gives them everything they need to build a complete, playable version. No format is required. Write the spec however you think will best serve the developer. Be as detailed as you think necessary.

5 planners (Opus, GPT-5.4, Gemini Pro, GLM-5, and a no-planner control). 8 builders. 10 games. 400 total builds.

Automated Playwright QA — browser-based gameplay testing that sends game-specific inputs, takes screenshots, and detects frozen/blank/broken states.

2.3 Model Narratives

Claude Sonnet 4.6 — The reliable workhorse. 9.35 average across all V1 runs, zero DNFs. Doesn't always hit the highest peaks but never bottomed out. Best all-round choice for autonomous coding.

GPT-5.4 — The hard-game specialist. Tends to slightly outperform others on complex titles (Pac-Man, Donkey Kong) while matching Sonnet on simpler ones. Best for when you need peak quality and have time to review.

Claude Opus 4.6 — Capable but context-sensitive. Strong individual scores when runs stayed short, but accumulated context across 10 games caused session failures. Not a reliable solo builder at scale.

Claude Haiku 4.5 — Spec-dependent. Strong with concise plans, weaker with detailed ones. Budget option that punches above its weight when you give it the right inputs.

Gemini 3 Flash — The value champion. Cheap, fast, and consistently scored near the top — often beating Pro. Gemini Pro had catastrophic outliers; Flash was stable across all game types.

o3-mini — Confabulates. Reports tasks complete that were never built. Any benchmark including o3-mini needs a completion verification step. Not suitable for autonomous code generation.

2.5 What V1 Got Wrong

Critical methodology flaws — what we'd change

Template contamination: Both Opus and GPT-5.4 filled in the same 12-section form. Neither was free to design the spec. The "planner comparison" measured form-filling, not genuine planning quality. This directly led to Round 2's unconstrained design.

Boilerplate task injection: Task T11 in the template was "implement lives/health indicator" — verbatim across all 10 games, including Pong, Tetris, and Snake, which have no lives system. Small builders spent cycles on irrelevant requirements.

Static QA misses runtime bugs: The 3-judge panel gave 7–9/10 Playability scores to games with movement-freezing bugs. Human play and browser agent both caught what static review missed. This is why Round 2 used automated playtesting.

o3-mini confabulation: Any benchmark including o3-mini needs a completion verification step. Never trust the reported task status.

Part 3 — Round 2: Planning From Scratch

V2 400 builds · 5 planners · 8 builders · Automated playtest QA

3.1 What We Changed

Round 2's central question: Does planning quality affect build quality when the planner is unconstrained?

No template. No section headings. No imposed structure. Each planner received only a brief and was told to write the spec however they thought best. The builder then received the spec verbatim — no modifications.

We also added a control group: games built with no spec at all (the builder received only the game name). This gave us a baseline for "no structure" to compare against.

3.5 The Opus Paradox

"The best planning model produces the worst builds. Opus 4.6 specs — with their pixel-perfect coordinates and exact physics constants — consistently underperform specs written by weaker models, and dramatically underperform no spec at all."

— Round 2 key finding

Opus 4.6 is widely considered the best AI model for complex reasoning tasks. As a builder, it ranked #2 (6.62). As a planner, it ranked last across all 5 planners (4.90) — below even Gemini Pro's 93-line sketch.

The difference: GLM-5's specs were long and vague, full of high-level descriptions. Opus's specs were long and precise, with exact coordinates that left no room for the builder to interpret. 4.3 The Planning Paradox — Full Picture

"Opus is better at filling templates than at unguided planning. In Round 1 (constrained), Opus specs gave a marginal +0.12 advantage. In Round 2 (unconstrained), they produced a −1.69 disadvantage."

— Why V1 and V2 contradict each other

In V1, both Opus and GPT-5.4 used the same template — they were essentially filling in the same form. Opus filled it in more thoroughly, producing slightly better results. In V2, both were unconstrained — and Opus's thoroughness became a liability. Its precise specifications constrained the builder's ability to apply their own judgement.

Part 5 — Practical Guide: Setting Up AI Agents for Coding Tasks

5.1 Model Selection

Use caseRecommendationWhy
Ship quality, no supervisionClaude Sonnet 4.6Most reliable. Zero DNFs across 182 builds. Doesn't spike highs but never crashes.
Budget builds, needs reviewGemini 3 Flash or GPT-5.4 NanoBest score-per-dollar. Flash especially stable — fewer outliers than Pro.
Hard problems (Pac-Man, DK)GPT-5.4Best peak performance on complex games. Worth the cost when quality matters most.
Fast prototypingClaude Haiku 4.5Good with concise specs. Autonomy level depends on spec quality.
Writing specs/plansGemini 3.1 ProShort, behaviour-focused specs. Less over-specification damage than Opus.
Avoid — autonomous codingo3-miniConfabulates. Never trust reported task completion status.
Avoid — as planner for weak buildersClaude Opus 4.6Over-specification hurts weaker models. Fine as a builder; avoid as spec author for smaller models.

5.2 Spec Writing Best Practices

  1. Match detail to builder capability. Frontier models handle ambiguity; value models need clarity. Don't give Gemini Flash a two-sentence brief and expect a perfect Donkey Kong.
  2. Avoid pixel-exact coordinates. Use relative or behavioural descriptions. "The player starts at the bottom" is better than "Player spawns at (336, 720)".
  3. Prose > formulas for smaller models. "The ball bounces at the same angle it arrived" is clearer than "reflect(v, n) where n is the surface normal".
  4. Game-specific task lists > generic templates. Don't include "implement a lives system" for Tetris. Irrelevant boilerplate wastes token budget and confuses smaller builders.
  5. Fewer constraints = more room for the builder's strengths. If you're unsure whether detail helps or hurts, err towards less. The no-planner control (6.59) beat all five planners.

5.3 Infrastructure Lessons

  • Isolated context per task. Opus's DNFs in V1 were an infrastructure problem, not a capability problem. Give each build a clean session.
  • Build verification step. Always verify output exists and is non-trivial before declaring success. o3-mini reported tasks complete that were never built.
  • Automated playtest QA > static code review. The 2-point gap between static and playtest scores means you need to actually run the code.
  • Per-build timeouts. Playwright can hang indefinitely on broken builds. Set explicit timeouts (45s per build worked well for us).
  • Workspace contamination. Bench agents should not load your MEMORY.md or AGENTS.md. We saw ~5K tokens of irrelevant context bleeding into every build session.

5.4 What We'd Do Differently

Methodology improvements for the next round
  1. Clean workspace for bench agents. Remove SOUL.md, MEMORY.md, AGENTS.md from the context that bench agents load. Our benchmark scores may be slightly depressed by the ~5K tokens of irrelevant workspace context.
  2. Human playtest sample alongside automated QA. Automated testing catches crashes but can't judge whether a game "feels right". A small human sample would calibrate the automated scores.
  3. Multiple builds per model×game. Single builds have high variance. 2–3 builds per cell would give statistical confidence.
  4. Planner specs reviewed for correctness before builder phase. A quick sanity-check that the spec actually describes the right game would catch fundamental mismatches before they propagate to 8 builders.
  5. Output token limit testing. Did any builds hit the maxTokens ceiling? Checking would confirm whether token limits are a bottleneck.

Appendix C — Methodology Notes

V1 (March 2026): Orchestrating agent generated all specs using a 12-section template. Two spec authors (Opus, GPT-5.4). 3-judge static QA panel scoring 5 dimensions (1–10 each). 8 runs × ~10 games × multiple builders.

V2 (April 2026): 5 planners given only a brief. No template. Specs passed verbatim to builders. Playwright automated playtesting. 5 runs × 10 games × 8 builders = 400 builds. QA scored 0–10 per build.

Limitations: Single builds per cell (no statistical replication). Automated QA may miss subtle gameplay issues. Workspace context contamination in bench agents. Control group (r5) only 29% QA'd at time of analysis.
