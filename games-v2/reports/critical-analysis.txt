SECTION 1: WHAT WAS PROMISED VS WHAT WAS DELIVERED
================================================================================

PROMISED: 5 runs × 10 games × 8 builders = 400 builds
DELIVERED: 354 complete builds

Breakdown:
  r1-opus:         74 complete, 5 DNF, 1 missing = 80 (80% complete builds)
  r2-gpt54:        70 complete, 10 DNF = 80 (87.5% complete)
  r3-gemini-pro:   70 complete, 4 DNF, 6 missing = 80 (87.5% complete)
  r4-glm5:         70 complete, 0 DNF, 10 missing = 80 (87.5% complete)
  r5-control:      70 complete, 10 DNF = 80 (87.5% complete)

Total: 354 complete, 29 DNF, 17 missing = 400 entries
Missing/DNF rate: 46/400 = 11.5% of theoretical builds never completed.

PROMISED: Static QA (3-judge panel) scoring all builds
DELIVERED: 0 static QA reports on disk

The file scripts/03_static_qa.sh exists and was never run.
All final_score values in results.json are null until we computed proxy scores
from browser QA errors — which were themselves incomplete (see below).

PROMISED: Browser QA on all complete builds
DELIVERED: Incomplete

Browser QA reports on disk (per run):
  r1-opus:         74 reports  (matches complete count — FULL)
  r2-gpt54:        70 reports  (matches complete count — FULL)
  r3-gemini-pro:   70 reports  (matches complete count — FULL)
  r4-glm5:         70 reports  (matches complete count — FULL)
  r5-control:      20 reports  (NOT FULL — only Sonnet + GLM-5)

r5-control: 70 complete builds, only 20 QA reports exist.
  QA'd: bench-sonnet (10) + bench-glm5 (10)
  NOT QA'd: bench-opus, bench-haiku, bench-gpt54, bench-gemini-pro, bench-gemini-flash
  → 60 complete r5 builds have NO browser QA data whatsoever.

Root cause: qa_browser_v2.py originally had RUNS = ["r1-opus", "r2-gpt54",
"r3-gemini-pro", "r4-glm5"] — r5-control was missing from the list.
A later manual fix added r5 to RUNS, but only Sonnet QA was re-run.

PROMISED: 8 builders tested against each planner
DELIVERED: Inconsistent

r4-glm5: bench-gpt54mini was skipped (--skip-builder bench-gpt54mini in the build
command). 10/80 potential r4 builds were never generated. This was intentional
but undisclosed in results. The GLM-5 planner row therefore reflects only 7
builders, not 8 — removing the worst-performing builder from its results
artificially inflates GLM-5's average.

r3-gemini-pro: bench-gpt54mini had 6 missing + 4 DNF. Only r1 had scored
Mini builds.

PROMISED: GPT-5.4 specs tested as a planning model
DELIVERED: Ambiguous

The r2-gpt54 planning run used GPT-5.4 FLAGSHP as the spec writer, not Mini.
But in the builder dimension, GPT-5.4 Mini was tested (and failed catastrophically).
GPT-5.4 Mini as spec writer was NEVER tested — it wasn't a planning model.
The experiment conflates "GPT-5.4 specs" (flagship-written) with "GPT-5.4
builder" (which includes Mini), making the planner-vs-builder comparison
meaningless for this model family.

================================================================================
SECTION 2: EXECUTION FAILURES
================================================================================

FAILURE 1: QA script RUNS list was wrong
The QA script excluded r5-control from its RUNS list. This was discovered and
partially fixed, but the fix was incomplete — only Sonnet was re-run on r5.
60 builds remain permanently un-QA'd.

FAILURE 2: No skip status tracked in results.json
The build log contains 140 SKIP entries (builds the orchestrator chose not to
attempt). These don't appear as "skipped" in results.json — they show as
"missing". There is no way to distinguish "build failed" from "build was
intentionally skipped" without cross-referencing the build log.

FAILURE 3: Static QA was promised and never delivered
03_static_qa.sh exists but was never executed. results.json has no judge scores.
The entire scoring methodology is based on a single automated QA pass (Playwright
browser testing) rather than the multi-judge panel described in the methodology.

FAILURE 4: Scoring formula is post-hoc, not validated
Score = 10 - min(errors, 10). This formula was invented after the fact when
static QA failed. It has never been validated as a proxy for game quality.
Builds with 11 errors and 150 errors both score 0 — catastrophic failures
are indistinguishable from mildly buggy ones.

================================================================================
SECTION 3: DATA INTEGRITY ISSUES
================================================================================

ISSUE 1: GLM-5 planner results are biased
r4-glm5 intentionally excluded GPT-5.4 Mini (--skip-builder bench-gpt54mini).
The GLM-5 planner was therefore tested against 7/8 builders, always skipping the
weakest builder. No other planner had this advantage. GLM-5's clean DNF record
is partly an artifact of excluding Mini, not purely a property of its specs.

ISSUE 2: GPT-5.4 Mini data is too sparse for conclusions
GPT-5.4 Mini theoretical builds: 50 (5 runs × 10 games)
  - Complete (scored): 10 (r1 only) — avg 9.10
  - DNF: 24 (r2: 10, r5: 10, r3: 4)
  - Missing/skipped: 16 (r3: 6, r4: 10)
Data coverage: 10/50 = 20% of Mini builds scored.
Any conclusion about GPT-5.4 Mini as a builder is based on 1/5 runs.

ISSUE 3: r5 control is incompletely QA'd
60/70 r5-complete builds have no browser QA data. 
The r5 "control" comparison (no specs vs specs) is therefore only partial.
We can say r5 Sonnet had catastrophic failures on asteroids/galaga, but we
cannot say anything meaningful about r5 Opus, Haiku, GPT-5.4 flagship, Gemini
Pro, or Gemini Flash in the no-specs condition.

ISSUE 4: No statistical significance testing
No margin of error, confidence intervals, or p-values reported. With n=8 builder
comparisons per planner and n=5 planner comparisons per builder, many observed
differences (e.g., GPT-5.4: 9.94 vs Haiku: 9.90) are within noise range.

ISSUE 5: Score distribution is bimodal and uninformative
Of 354 complete builds:
  - 274 score 10.0 (zero JS errors — 77%)
  - 47 score 9.0-9.9 (1-10 errors)
  - 33 score 0.0 (10+ errors, capped)
The distribution clusters at extremes. The 0-10 scale compresses everything
above 10 errors into a single score.

================================================================================
SECTION 4: METHODOLOGICAL PROBLEMS
================================================================================

PROBLEM 1: Builder identity conflation
"GPT-5.4" is used to mean both GPT-5.4 flagship (powerful, 0 DNFs) and
GPT-5.4 Mini (weak, 24 DNFs). These are different models with different
capability profiles. The experiment design tests them together as one "builder"
in the matrix, which masks the flagship's excellent performance and the Mini's
catastrophic failure under a single row.

PROBLEM 2: Planning model coverage is incomplete
5 planners were tested, but:
  - GPT-5.4 Mini as spec writer: NEVER TESTED
  - GLM-5 as spec writer: tested in r4 only
  - We don't know if Mini can write specs at all (no rX-mini planning run)
  - We don't know if GLM-5 specs are robust across all builder types beyond r4

PROBLEM 3: No counterbalancing
The experiment uses a crossed design (5 planners × 8 builders) in theory, but
in practice: r4 skipped Mini, r3 had Mini failures+skips, r5 only has 2/8
builders QA'd. The design is nearly orthogonal but execution broke it.

PROBLEM 4: Missing test for spec quality × builder quality interaction
The central question ("do better specs produce better games?") requires
controlling for builder quality. We cannot answer it properly because:
  (a) Some builders (Mini, Sonnet) fail catastrophically regardless of spec
  (b) Some runs (r5) only have partial QA
  (c) GLM-5's results are biased by Mini exclusion

PROBLEM 5: No baseline validation
We never established that the scoring methodology (browser QA errors) correlates
with human judgment of game quality. The V1 report explicitly noted that static
QA gave games 7-9/10 while browser QA found movement bugs that humans noticed
immediately. Proxy scoring from error counts has not been validated.

================================================================================
SECTION 5: WHAT ACTUALLY CAN BE CONCLUDED
================================================================================

The following conclusions are SUPPORTED by the data:

1. GLM-5 specs appear to produce fewer catastrophic failures than Opus or no
   specs, at least for the 7 builders that were tested with them. DNF rate: 0%
   for GLM-5 vs 6-12.5% for others.

2. GPT-5.4 flagship is the best builder tested: 50/50 completions, 0 DNFs,
   9.94 avg score. Consistent across all 5 runs.

3. Sonnet 4.6 is spec-dependent: 3 catastrophic DNFs in r1 (Opus specs) with
   150+ JS errors, but 0 DNFs in r2-r5. Requires good specs to not fail badly.

4. GPT-5.4 Mini is unreliable as a builder: 24/50 DNFs across the 3 runs it
   was attempted in. Data too sparse to characterize fully.

5. No specs produces more zero-JS-error builds than GPT-5.4 specs (68/70 vs
   59/70), but the same DNF rate (12.5%). Generic code is clean when it works,
   equally broken when it doesn't.

6. Pac-Man is the hardest game to build: lowest avg scores across all builders
   and planners. Complex state management defeats AI implementations.

The following CANNOT be concluded from this data:

- Whether GLM-5 specs are genuinely better than Gemini Pro specs (score difference
  is 9.93 vs 9.96, within noise)
- Whether GPT-5.4 specs are better or worse than no specs (DNF rates are
  identical at 12.5% but r5 is only 20% QA'd)
- Whether Haiku (9.90) is meaningfully different from GPT-5.4 flagship (9.94)
- Anything about GPT-5.4 Mini as a spec writer (never tested)
- Whether the observed differences are stable or noise

================================================================================
SECTION 6: SUMMARY
================================================================================

The Benchmark V2 framework had significant execution failures that prevent clean
answers to the central research question. Specifically:

1. Static QA never ran — scoring is based on unvalidated browser QA proxy
2. r5-control only 29% browser QA'd (20/70 complete builds)
3. GPT-5.4 Mini data is 80% missing — any builder conclusions are unreliable
4. GLM-5 results are biased by intentional Mini exclusion
5. The "5×8 crossed design" was only approximately achieved
6. Skip vs DNF vs missing is not tracked consistently in results.json

The data that was collected is informative but incomplete. The framework cannot
support strong claims about spec quality effects without re-running the missing
QA runs and completing the static QA pipeline.

================================================================================
SECTION 7: ADDITIONAL FINDING — GPT-5.4 MINI FRAMEWORK LIMITATION (post-audit)
================================================================================

Post-analysis investigation revealed:

The bench-gpt54mini agent is configured with openai/gpt-5.4-mini on OpenRouter.
However, the OpenClaw system prompt for this agent is 44,849 characters:
  - Workspace files: ~23K chars (AGENTS.md, SOUL.md, MEMORY.md, etc.)
  - Tool schemas: ~27K chars
  - Skills loaded: ~8K chars

When combined, this exceeds GPT-5.4 Mini's context handling threshold on OpenRouter.
The gateway rejects the prompt preemptively before any API call is made.

Evidence:
- Session log for r6-mini spec generation: assistant message has no content, finish_reason=null
- Direct API test via palebluedot: model ID "openai/gpt-5.4-mini" not in available models
- OpenRouter test: 44K+ system prompt triggers preemptive context overflow
- Tokens observed: 0 input, 0 output — no API call was made

CONCLUSION: GPT-5.4 Mini cannot operate as an OpenClaw agent due to the framework's
system prompt overhead. This is NOT a model capability issue — it is a framework
configuration issue. Mini could potentially write specs if given a minimal agent
profile (no workspace files, no skills, minimal tools), but that configuration was
not available at audit time.

IMPACT ON V2 CONCLUSIONS:
- r6-mini planning run: CANNOT PROCEED (framework limitation, not model failure)
- GPT-5.4 Mini as spec writer: UNTESTED
- The experiment design gap (Mini never tested as spec writer) is now explained by
  framework constraints, not by any inference about Mini's capabilities
- The 24 DNFs from Mini as builder remain valid data points — Mini clearly cannot
  build games reliably even when receiving properly formatted prompts
